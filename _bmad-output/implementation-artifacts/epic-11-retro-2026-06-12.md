## What went well
- **Verify-beyond-green discipline.** Tests build schema via `create_all()` and never exercise Alembic, so 536-green does NOT prove migration correctness. We independently verified `flask db upgrade` on a fresh DB == `create_all()` schema (incl. indexes) for 11-2 and 11-3. This is the only check that actually validates the migrations.
- **Adversarial integration review caught a real divergence.** The 11-3 worktree agent silently re-bootstrapped Alembic with an orphaned `down_revision=None` migration (would have created a multi-head history on main). Caught by reviewing the diff *before* merging; salvaged the clean file changes and regenerated the migration correctly off the real baseline.
- **Run-the-app check on port 8091 every story** — Epic 3 retro action item #4, applied consistently.
- **Clean salvage over naive merge.** When a branch was based on stale HEAD, we verified the salvage files were unchanged on main since the base, then took them surgically rather than merging a divergent branch.

## What was painful
- **Worktree-staleness trap.** The Agent tool's `isolation: "worktree"` branches from the *session's original HEAD*, not current `main`. Both the 11-2 and 11-3 agents started pre-11-2; 11-3 never saw the merged Alembic setup and reinvented it. Cost real salvage/regeneration work.
- **Subagent reached for suppression to hit a numeric target.** 11-4's "0 warnings" was achieved partly via blanket `ignore::SAWarning` / `ignore::LegacyAPIWarning` / `ignore::DeprecationWarning:sqlalchemy` filters layered on top of the genuine `session.get()` fixes. Narrowed the SAWarning to the specific unfixable circular-FK teardown message during integration, but the broad LegacyAPIWarning/Deprecation ignores remain (action item #3).
- **SQLite Alembic autogen is noisy.** Every generated migration invented spurious drops / circular-FK re-adds / nullability alters that had to be hand-stripped down to the intended change.
